## Original Monolith (`spring-music/`)

### Build & Run

```bash
# Build
./gradlew clean assemble
# Output: build/libs/spring-music-1.0.jar

# Run tests
./gradlew test

# Run a single test class
./gradlew test --tests org.cloudfoundry.samples.music.ApplicationTests

# Run locally (default H2 in-memory)
java -jar build/libs/spring-music-1.0.jar

# Run with a specific database profile
java -jar -Dspring.profiles.active=postgres build/libs/spring-music-1.0.jar
```

### Architecture

Spring Music is a sample CRUD app that demonstrates pluggable databases via Spring Data. A single `Album` domain object is persisted through one of three repository implementations chosen by the active Spring profile.

**Profile → Repository mapping:**

| Profile | Repository | Storage |
|---|---|---|
| default (none) | `JpaAlbumRepository` | H2 in-memory |
| `mysql` / `postgres` / `oracle` / `sqlserver` | `JpaAlbumRepository` | Relational DB |
| `mongodb` | `MongoAlbumRepository` | MongoDB |
| `redis` | `RedisAlbumRepository` | Redis hash |

Only one database profile may be active at a time. `SpringApplicationContextInitializer` enforces this, auto-detects bound Cloud Foundry services by tag, activates the matching profile, and dynamically excludes the Spring Boot auto-configurations that aren't needed.

**Startup sequence:**
1. `SpringApplicationContextInitializer` runs before beans are created → reads CF services → activates profile → excludes irrelevant auto-config
2. Active `@Profile`-annotated repository bean is registered
3. `AlbumRepositoryPopulator` fires on `ApplicationReadyEvent` → seeds `albums.json` into the repository if it is empty

**Key classes:**

| Class | Role |
|---|---|
| `Application` | Entry point; registers the initializer and populator |
| `Album` | Domain model — JPA `@Entity`, MongoDB document, Redis hash value; ID generated by `RandomIdGenerator` |
| `AlbumController` | `GET/PUT/POST/DELETE /albums` — delegates entirely to `CrudRepository` interface |
| `InfoController` | `GET /appinfo`, `GET /service` — reads active profiles and CF-bound service metadata via `CfEnv` |
| `ErrorController` | `GET /errors/kill|fill-heap|throw` — intentional crash endpoints for CF self-healing demos |
| `SpringApplicationContextInitializer` | CF service detection + profile activation + auto-config exclusion |
| `AlbumRepositoryPopulator` | One-time seed loader from `albums.json` |
| `RedisConfig` | `RedisTemplate<String, Album>` bean with Jackson serialization (active on `redis` profile only) |

**REST endpoints:**
- `GET /albums` — list all; `GET /albums/{id}` — single; `PUT /albums` — create; `POST /albums` — update; `DELETE /albums/{id}` — delete
- `GET /appinfo` — active profiles + bound CF service names
- `GET /service` — full CF service binding details
- `GET /actuator/*` — Spring Boot actuator

### Configuration

- `src/main/resources/application.yml` — base config + profile-specific datasource blocks (uses old `spring.profiles:` YAML key, valid for Spring Boot 2.4)
- `src/main/resources/albums.json` — seed data; `_class` field is used by MongoDB, ignored by JPA/Redis
- `manifest.yml` — CF manifest: 1 GB, random route, auto-reconfiguration disabled

### Cloud Foundry Deployment

```bash
./gradlew clean assemble
cf push

# Bind a service — profile is auto-selected from service tags
cf create-service <service> <plan> <name>
cf bind-service spring-music <name>
cf restart spring-music
```

## Modernized Codebase (`../spring-music-new/`)

Three independently deployable Spring Boot 3.2.3 / Java 17 services. Each has its own `build.gradle`, `gradlew`, and `manifest.yml`.

### Services

| Service | Port | Responsibility |
|---|---|---|
| `album-catalog-service` | 8080 | Album CRUD, all DB logic, seed loading, CORS, exception handling |
| `app-info-service` | 8081 | `/appinfo` + `/service` metadata endpoints |
| `spring-music-ui` | 8082 | Static AngularJS SPA + reverse proxy to the other two services |

The `ProxyController` in `spring-music-ui` forwards `/albums/**` and `/errors/**` to the catalog service, and `/appinfo` + `/service` to the info service — keeping all AngularJS relative URLs unchanged.

### Build & Run (each service)

```bash
cd album-catalog-service   # or app-info-service / spring-music-ui
./gradlew clean assemble
java -jar build/libs/<service>-1.0.jar

# Run all three locally
# Terminal 1: album-catalog-service  (default H2)
# Terminal 2: app-info-service
# Terminal 3: spring-music-ui  →  open http://localhost:8082

# Tests
./gradlew test
./gradlew test --tests '*.ApplicationTests'
```

### Key Modernization Changes vs Original

- `javax.*` → `jakarta.*` (Spring Boot 3 / Jakarta EE 10)
- `spring.profiles:` → `spring.config.activate.on-profile:` in YAML
- MySQL driver: `mysql:mysql-connector-java` → `com.mysql:mysql-connector-j`
- Explicit Hibernate dialects removed (Hibernate 6 auto-detects)
- `RandomIdGenerator` updated to Hibernate 6 `SharedSessionContractImplementor` API
- `ApplicationInfo` converted from mutable POJO to Java 17 `record`
- `@RequestMapping` → `@GetMapping`/`@PostMapping` etc.; HTTP 201/204 for create/delete
- `GlobalExceptionHandler` (`@RestControllerAdvice`) returns RFC 7807 `ProblemDetail`
- `WebConfig` adds CORS for the UI service origin
- Build: `plugins {}` DSL, `mavenCentral()` only (jcenter removed), Gradle 8.7
